Brief

 

Existing Apps

A search for suitable trail camera photo classifiers has not turned up any good apps. We want to monitor wetland birds.

 

The ZIP Classifier is free, but not open, is set to expire, only works on a Window OS and does not link to the camera locations. I could not get it to work reliably. The ZIP idea of renaming the images with the common species name and date, sorting into different folders by name is very eccentric. A CSV catalog is primitive. Surely we can do better?

 

TrapNZ does have camera monitoring points and image records, but the classifier is very limited. All the images have to be uploaded and video images are not supported. No comments are provided. We could use the TrapNZ mobile app to record camera locations to save having to use a GPS device or build a Survey123 app.

 

Catalog

We need a robust catalog database of the images linked to the camera locations so that we can show the incidence of detected species just like other monitoring tools.  We need a custom list of target species with official names that can be easily amended. The results need to be able to be reviewed and edited. Perhaps put the classified metadata back into the image as well as a database and keep the image name as the original. Adding any note seems to be essential for special cases.

 

Classification of Species

The different lists of target species is very disappointing, not using proper names that are standardised across all ecology. It should use the NZ Organism Reference NZOR names for interoperability. This is easy to set up, it is free to download and filter.

 

Multiple targets

Trialling the other two apps shows that we need to be able to tag more than one species in a shot and count multiple instances of the same species. ZIP Classifier did this, TrapNZ did not.  A quick way of tagging requires an active table of choices. ZIP uses one or two keys or a pointer, picking a list is easier and quicker for complex classification.

 

Speed

Often there are a lot of images. Being able to quickly scan the set for useful images is important. A slideshow function helps quickly step through the photos. Each event can be set up to take a still photo and then a short video. It may be better to split images and videos into  separate folders.

 

Videos and storage

The cameras can take still and video shots so it has to be able to handle both.

We need to be able to retrieve best shots for printing and publicity. Video images are a 10 second run after being triggered but that is often far too long. A way of truncating the video would be useful. Images with no targets should be deleted.

 

Where should the images be stored? Each camera has its own numbering of images so we could get clashes. What sort of folder structure? Should they be in the cloud? How do we keep the link between the catalog and the image.

 

Mapping and Classification

If we use QGIS there are good tools to build forms, display images and videos quickly and store the metadata in a geopackage sqlite database. The location of the cameras is important so we need a point layer of camera positions. Multiple images are associated with a single camera site so we need a related table that has image attachments.

 

Graphical Results

The purpose of the monitoring is to act on the results found. We need some maps and serial charts to show the distribution over time of the species being monitored. A spreadsheet is just not good enough. QGIS already has tools built-in to produce graphical output.

 

GitHub

The application code is hosted on GitHub so it may be adopted by others. The entire app can be packaged as a QGIS Plugin. The species lookup table is editable as is the catalog table template. The development environment in QGIS would be a powerful way of creating a responsive menu while still linking into QGIS maps, TrapNZ and AGOL.
